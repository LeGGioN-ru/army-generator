# army-generator


Из паттернов/подходов использовались.
1. В проекте использовался Zenject, дабы упростить инициализацию, в будущем может использоваться для подмены реализаций под интерфейсами (DI). 
2. Object Pool (Чтобы постоянно не инстанциироват/удалять объекты. Тем самым повышаем производительность).
3. Стейт машина - Т.к. у нас уже достаточно много состояний (идти до врага, искать врага, атаковать врага). Дабы код в кашу не превращался - разбиваем на стейты и транзишены. Транзишены реализуем структурами, дабы попроще для памяти было их обрабатывать.
4. Не чистый MVP - для солдата. Не чистый - имею ввиду не чистый C#, а с примесью монобехов, дабы не переусложнять работу/дебагинг/инициализацию в Юнити. Ну и MVP позволяет удобно разграничить данные и не перемешивать логику визуала, с логикой модели, с логикой взаимодействия. + позволит в будущем грузить модели из другого источника (локальный json, бэкенд)
5. Фабрика - Т.к. обычного инстантиейта не достаточно и нужно место, где будем инициализировать/создавать наших солдат.
6. Сигнальная шина - уже встроена в Zenject Framework. Удобно прокидывать глобальные события и отслеживать логику поведения приложения + подписываться на события дабы реагировать на поведение приложения.
7. Реализация статов и их отображения - разделена. У нас есть модели которые отвечают за сами статы, которые мы удобно можем докидывать и убирать. У нас есть модели для вьюшек (цвет, форма, размер). И сами вьюшки которые принимают StatView, в которых уже есть Stats. В будущем позволит удобно переиспользовать Stats, для каких-то бафов и прочего.
8. Интерфейс ISoldier сделан, дабы не прокидывать Solder Presentor целиком, я бы разделил интерфейс ещё больше (Interface segregation), но по контексту задачи - этого должно быть достаточно.
9. Доп фича игры - ставки. Гемблинг нынче востребован.
